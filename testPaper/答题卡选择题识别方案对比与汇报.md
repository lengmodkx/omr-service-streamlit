## 一、项目目标

**识别答题卡中选择题的填涂区域，自动获取每题答案。**

当前面对的问题：
- 不同科目试卷排版不同（英语、物理等），选项数量从 3 到 7 个不等
- 同一科目后续可能更换试卷格式，需持续适配新排版
- 扫描仪批次不同，光照、偏移、旋转存在差异

---

## 二、OpenCV 传统方案（已完成尝试）

### 2.1 尝试过的三种方法

| 方法 | 原理 | 实验结论 |
|------|------|----------|
| **模板坐标 + 像素暗度阈值** | 手动标注每个气泡的 (x,y,w,h)，缩放到图片后检测中心像素暗度 | 对坐标偏移极度敏感，光照一变阈值就失效 |
| **差分法（空白卡对比）** | 将填涂卡与空白卡做像素差分，暗度变化大的判定为填涂 | 需要同一扫描仪、同一批次空白卡，实际场景难以保证 |
| **自动网格检测** | 用户画出每列大致区域 + 行列数，系统自动生成检测点网格，blob检测判断填涂 | 对不同排版需要重新画网格，且blob参数在不同印刷质量下不稳定 |

### 2.2 OpenCV 方案的核心问题

| 问题 | 具体表现 | 根因 |
|------|----------|------|
| **坐标偏移** | 模板坐标缩放到实际图片后，气泡位置偏 3-8 像素 | 扫描仪机械公差、纸张伸缩、页面倾斜 |
| **阈值不稳定** | 同一阈值在图片A准确，图片B大量漏检/误检 | 光照环境、打印机墨粉浓度不同 |
| **题号干扰** | 选择题题号数字被判定为填涂气泡 | 传统CV无法区分"圆形填涂块"和"数字文字"的视觉特征 |
| **新排版成本高** | 每种新格式需重新标定坐标/网格 | 参数化方法依赖人工配置 |
| **准确率天花板** | 多次调优后仍在 80%~85%，无法稳定达标 | 规则式方法在多变场景下的天然局限 |

> **结论：OpenCV 方案在单一固定格式 + 严格受控扫描环境下可用，但在多格式、多批次、实际生产环境中无法满足准确率要求。**

## 三、YOLO 深度学习方案

### 3.1 核心思路

```
传统OpenCV：         模板坐标 → 缩放 → 像素暗度判断 → 易受偏移/光照干扰

YOLO方案：           YOLO检测填涂标记(mark) → 匹配到模板气泡 → 确定题号+选项
                         ↑                            ↑
                    视觉特征学习                    坐标映射（鲁棒）
```

**只训练一个类别 `mark`（填涂标记）**，让 YOLO 学习"涂黑的圆点/方块长什么样"。检测到 mark 后，按最近距离匹配到模板中的气泡坐标，确定是哪道题的哪个选项。

### 3.2 为什么 YOLO 能解决 OpenCV 的问题

| 问题 | YOLO 如何处理 |
|------|--------------|
| 坐标偏移 | 不依赖像素级定位，检测框有天然的空间容忍度 |
| 阈值不稳定 | 学习的是"涂黑"的**视觉特征**（颜色、纹理、形状），不是固定灰度阈值 |
| 题号干扰 | 数字和填涂块外观完全不同，YOLO 可通过训练样本学会区分 |
| 新排版 | 只需标注 **少量新格式样本**（~20张）加入训练，无需改动代码 |
| 旋转/倾斜 | YOLO 对几何变换具有天然鲁棒性（数据增强 + 卷积特性） |

### 3.3 预期效果

| 指标 | OpenCV 方案 | YOLO 方案 |
|------|:--------:|:------:|
| 选择题识别准确率 | ~80-85% | **> 95%** |
| 新增试卷格式成本 | 重新标定坐标（半天） | 标注20张新样本（1-2小时）+ 微调训练（10分钟） |
| 光照/偏移容忍度 | 低 | **高** |
| 推理速度 | < 0.5秒/张 | < 0.5秒/张 |

## 四、YOLO 部署方案对比：本地训练 vs 阿里云训练

### 4.1 方案概述

| | 方案A：本地训练 | 方案B：阿里云训练 |
|------|---------------|------------------|
| **训练环境** | 本地 RTX 4060 (8G) | 阿里云 GPU 实例 (V100) |
| **标注工具** | Label Studio（开源免费） | iTAG 自助标注（免费） |
| **数据安全** | 数据不出本地 | 需上传至云端 |
| **推理部署** | 本地 Streamlit / FastAPI | 阿里云 ECS 或 OCR自学习服务 |

### 4.2 详细对比

#### 方案A：本地训练（推荐）

**优点：**
- **零持续费用**：训练和推理均在本地，无 API 调用费、无云资源租金
- **数据安全**：答题卡含学生考号等信息，数据不离开内网
- **完全自主**：模型、代码、数据全掌控，不受云服务变更/下架影响
- **硬件已具备**：RTX 4060 8G 显卡完全满足 YOLOv8 训练需求
- **迭代灵活**：新增试卷格式可立即标注训练，无需走云资源申请流程

**缺点：**
- 需投入标注人力（初始约 800 个标记，10 人天）
- 团队需掌握基础 YOLO 操作（已有完整脚本支持）

**成本：**

| 项目 | 费用 |
|------|:----:|
| GPU 硬件（RTX 4060） | ¥2,000（已投入） |
| 标注人力 | 约 10 人天 |
| 训练电费 | < ¥10 |
| 推理调用 | ¥0 |
| **长期持有成本** | **约 10 人天 × 初次标注 + 零持续费** |

#### 方案B：阿里云训练

##### B1：GPU 实例自训练 YOLO

**优点：**
- 无需本地 GPU，上手即可训练
- 弹性计费，训练完释放实例

**缺点：**
- 推理仍需服务器（本地或云ECS），持续产生费用
- 数据需上传至云端
- 网络延迟影响批量处理效率

**成本：**

| 项目 | 费用 |
|------|:----:|
| GPU 训练 (V100 × 2h/轮 × 3轮) | ≈ ¥120 |
| iTAG 自助标注 | ¥0 |
| ECS 推理实例 (包月) | ¥400-600/月 |
| **年化成本** | **≈ ¥5,000-7,000** |

##### B2：阿里云 OCR 文档自学习服务

**优点：**
- 零代码，上传标注数据自动训练
- 阿里云 SLA 保障，服务稳定
- 模板类可直接输出结构化结果

**缺点：**
- 推理按次计费，规模越大成本越高
- 模型是黑盒，无法定制后处理（多选判断、异常检测等）
- 绑定阿里云平台，迁移成本高
- 自学习主要面向 KV/表格模板，答题卡气泡排列不一定是标准模板格式
